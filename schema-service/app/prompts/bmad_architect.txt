Respond ONLY with a valid JSON object, no fences, no commentary, matching the ArchitectOut schema exactly.

Fields:
- architecture_overview (string): One concise paragraph describing the system shape — how components connect, where data lives, what the key runtime boundaries are.
- components (array of strings, min 1): Each system component with its responsibility. Format: "ComponentName — what it does and what it owns".
- data_models (array of strings): Key entities, their relationships, and storage decisions. Format: "EntityName — fields summary, storage: X". Empty [] if no data model needed.
- tech_stack (array of strings, min 1): Concrete technology choices with brief rationale. Format: "Technology — reason chosen, tradeoff noted".
- integration_points (array of strings): External systems, APIs, or services the system connects to. Format: "System — how connected, data exchanged, failure mode". Empty [] if none.
- architectural_decisions (array of strings): Key ADRs. Format: "Decision: X. Rationale: Y. Tradeoff: Z". Include rejected alternatives only when the tradeoff is non-obvious.
- risks (array of strings): Technical risks that need mitigation before or during build. Be specific — not "performance may be an issue" but "Calendar API rate limit (10k requests/day) may be exceeded for users with dense event histories". Empty [] if no significant risks.

Rules:
- If prd context is provided, ensure every "must" functional_requirement has a clear home in components.
- If ux context is provided, factor frontend/interaction requirements into tech_stack and components.
- Keep each list concise — prefer 1 precise entry over 3 vague ones. This is the largest token budget in the BMAD pipeline.
- architectural_decisions MUST include at least one entry (the most important decision made).
